Our Recommendation

Alpaca Connect now. SnapTrade later.

Our current Alpaca setup (static Trading API key for market data) is a valid starting point, but the next step is migrating to Alpaca Connect (OAuth) to enable real per-user trading. Architecture Overview

Our platform operates as a thin orchestration layer between a user's bank account and a brokerage. We use two external services: Plaid for bank account linking and ACH funding, and a Brokerage Trading API for order execution and portfolio data retrieval. We do not hold funds, clear trades, or own any compliance obligations at this stage.

The correct API model for our use case is Trading API + OAuth/Connect, not a Broker API or BaaS platform. The distinction matters significantly for cost and liability:

  • Broker API model: Our company manages accounts on behalf of end users. We own KYC, pay per-account data fees, and carry compliance obligations. Costs $500–$2,000+/mo.
  • Trading API + OAuth model: Users have their own brokerage accounts. They authorize our app via OAuth 2.0 and we receive scoped tokens to trade and read data on their behalf. The broker owns KYC, custody, and compliance entirely. Our monthly infrastructure cost is effectively zero.
Our current Alpaca implementation uses a static API key (single paper trading account). The immediate next step is migrating to Alpaca Connect (OAuth) to enable real per-user accounts and trading. Apply for OAuth app access at alpaca.markets before beginning implementation — approval is required before production use.

Current Implementation Audit

The following files currently handle Alpaca integration in the app:

  • AlpacaConfig.kt — Static key pair via BuildConfig from local.properties. Used as auth headers on all requests. Contains stock and crypto symbol lists for market data fetching.
  • AlpacaClient.kt — Retrofit client using static key headers. Will need to be updated to accept per-user Bearer tokens from OAuth.
  • AlpacaApiService.kt — Endpoints for stock bars, crypto quotes, and assets. Order placement and portfolio endpoints not yet implemented.
  • StockTradingManager.kt — Orchestrates trading logic. Currently no real order placement — to be wired to per-user OAuth tokens.
  • SecureStorage.kt — Will be used to store per-user OAuth access and refresh tokens after migration.
⚠️ The current static key approach means all users share a single paper trading account. No real trades are possible and no user portfolio isolation exists. This must be resolved before any production trading feature ships.

Provider Technical Analysis

1. Alpaca — Trading API + Connect (OAuth) ✅ Recommended

Alpaca's Connect API is purpose-built for our architecture. Users create their own Alpaca accounts, authorize our app via OAuth 2.0, and we receive scoped access tokens to trade and retrieve portfolio data on their behalf. KYC is fully handled by Alpaca during user sign-up — we are completely out of that flow.

Authentication Flow

Standard OAuth 2.0 Authorization Code flow. Our app redirects users to Alpaca's authorization endpoint, the user consents, and we receive an access_token and refresh_token. All subsequent API calls use Bearer token auth scoped to that user's account. Tokens are stored securely in SecureStorage.kt per user.

Files to Create / Modify

  • New: AlpacaOAuthManager.kt — OAuth flow, token exchange, refresh logic
  • New: AlpacaConnectActivity.kt — WebView for OAuth login page with deep link callback
  • Modify: AlpacaConfig.kt — Add Client ID, redirect URI, OAuth scopes
  • Modify: AlpacaClient.kt — Swap static key headers for per-user Bearer token
  • Modify: AlpacaApiService.kt — Add /v2/orders, /v2/positions, /v2/account endpoints
  • Modify: StockTradingManager.kt — Wire up real per-user order placement
  • Modify: AndroidManifest.xml — Register clauapp://alpaca/callback deep link

Plaid Integration

Alpaca has an official first-party partnership with Plaid. After a user connects their bank via Plaid Link, our backend generates a Plaid processor_token specifying "processor": "alpaca" and passes it to Alpaca's ACH Relationships API. No custom bridge engineering required.

Assets & Endpoints

  • US equities (11,000+), ETFs, fractional shares — /v2/orders, /v2/positions, /v2/account
  • Options trading — /v2/options/contracts
  • Crypto 24/7 via Alpaca Crypto (FinCEN-registered) — same orders API, asset_class: crypto
  • WebSocket streaming — wss://stream.data.alpaca.markets
  • Market data — free IEX tier included; SIP tier ~$9/month for the platform

Pricing

Trading API access: free. No per-account fees. Commission-free on US equities. Crypto carries a small spread fee. Regulatory fees pass through as required by law.

Limitations

  • Users must create or already have an Alpaca account
  • We do not control the onboarding UX during Alpaca account creation
  • OAuth app approval required before production use — apply early
  • Rate limits: 200 requests/minute on free tier

2. Tradier — Trading API + OAuth 2.0 🟡 Consider

Tradier offers a clean OAuth 2.0 flow for third-party apps. KYC is handled entirely by Tradier. However, no crypto support and no official Plaid partnership make it a weak fit for our current product requirements.

Key Gaps

  • No crypto support — hard gap for our product roadmap
  • No official Plaid integration — custom ACH bridge engineering required
  • Partner approval adds lead time before OAuth is available in production

3. SnapTrade — Universal Brokerage Connector 🔵 Future Option

SnapTrade lets users connect existing accounts at 20+ brokerages (Fidelity, Schwab, Robinhood, Coinbase, etc.) through one API. Best suited for Phase 2 when onboarding data shows users dropping off at the "create an Alpaca account" step.

When to revisit: Once we have data showing meaningful drop-off at the Alpaca account creation step, SnapTrade solves that friction cleanly without requiring users to open a new account.

4. Interactive Brokers (IBKR) — Web API + OAuth 🔴 Defer

Broadest global asset coverage but heavyweight compliance approval, no Plaid partnership, complex session management, and requires funded Pro accounts. Significantly over-engineered for our current scope.

Revisit post-PMF if our roadmap requires global asset coverage (futures, forex, international equities) or institutional-grade execution quality.

Decision Framework

Use Alpaca Connect if:

  • We want stocks and crypto under one API with no extra integration work
  • We want the Plaid funding flow to work out of the box
  • We want zero monthly API cost at early scale
  • We are comfortable requiring users to create or already have an Alpaca account

Switch to or add SnapTrade if:

  • Onboarding data shows drop-off at the "create an Alpaca account" step
  • Users have strong existing relationships with Fidelity, Schwab, or Robinhood
  • We have sufficient volume to justify usage-based cost

Consider Broker API if:

  • We want to fully white-label the account creation experience inside Clau
  • We have funding to absorb $500–$2,000+/mo in platform fees plus per-user fees
  • We want to control KYC UX end-to-end within our app
Technical Recommendation

Migrate to Alpaca Connect. Build Plaid → ACH correctly. Revisit at scale.

Our current static key implementation is a valid MVP for market data only. The immediate priority is applying for Alpaca OAuth app access and migrating to per-user Bearer tokens. The highest-value engineering work after that is the Plaid processor token → Alpaca ACH Relationships flow, hardened with proper error handling, webhook reconciliation, and retry logic.

The next architecture decision will be driven by onboarding conversion data. If users drop off at Alpaca account creation, SnapTrade solves that cleanly. Neither that nor the Broker API is worth tackling before we have that signal.

Migration Overview

We are migrating from a single shared static API key to a per-user OAuth model. This unlocks real portfolio tracking, real trade execution, and a proper multi-user architecture. The migration has two hard prerequisites before a single line of code changes.

Prerequisites (Do These First)
Both must be completed before OAuth implementation can begin
Step 1 — Apply for Alpaca OAuth App Access

Go to alpaca.markets → Developer → OAuth Apps and submit an application. You will need your app name, description, redirect URI (clauapp://alpaca/callback), and requested scopes (account:write, data). You will receive a Client ID and Client Secret upon approval. This can take several days — apply immediately.

Step 2 — Add Client ID & Secret to local.properties

Once approved, add ALPACA_CLIENT_ID and ALPACA_CLIENT_SECRET to local.properties alongside the existing API keys. These are never committed to version control.

Implementation Roadmap
Phases to complete after OAuth credentials are received
Phase 1 — OAuth Flow
AlpacaOAuthManager + AlpacaConnectActivity
Implement the full Authorization Code flow. WebView redirects to Alpaca's authorize endpoint, captures the callback code via deep link (clauapp://alpaca/callback), exchanges it for access + refresh tokens, and stores them securely in SecureStorage per user.
Phase 2 — Per-User API Client
Update AlpacaClient + AlpacaConfig
Swap the static key headers in AlpacaClient for a per-user Bearer token injected at request time. AlpacaConfig gains OAuth constants (Client ID, redirect URI, scopes). Static keys remain as a fallback for market data only (unauthenticated users).
Phase 3 — Portfolio & Orders
Extend AlpacaApiService + StockTradingManager
Add /v2/account, /v2/positions, /v2/orders to AlpacaApiService. Wire StockTradingManager to place real orders and fetch real portfolio data per authenticated user.
Phase 4 — Plaid → ACH Funding
Plaid processor_token → Alpaca ACH Relationships
After user links their bank via Plaid Link, generate a processor_token specifying "processor": "alpaca" and pass it to Alpaca's ACH Relationships API to establish the funding link. Add error handling, webhook reconciliation, and retry logic. This is the highest-risk integration step and should be tested thoroughly in sandbox.
Phase 5 — Paper → Live
Switch users from paper to live trading environment
Alpaca supports both paper (paper-api.alpaca.markets) and live (api.alpaca.markets) environments. Users toggle between them via their Alpaca account. Our OAuth tokens automatically scope to whichever environment the user's account is authorized for.
Future — SnapTrade (If Needed)
Multi-broker support for users with existing accounts
If onboarding data shows drop-off at the Alpaca account creation step, SnapTrade eliminates that friction by letting users connect Fidelity, Schwab, Robinhood, and others. Evaluate after Phase 5 is live and conversion data is available.
